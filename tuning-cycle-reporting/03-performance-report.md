# 03. 성능 테스트 결과 보고서 — 비기술 직군을 위한 작성법

---

## 🎯 핵심 질문

"CEO나 PM이 이해할 수 있는 성능 보고서를 어떻게 작성할까?"

기술적 용어가 아닌 **비즈니스 언어**로 설명해야 의사결정이 가능합니다.

---

## 🔍 왜 이 개념이 실무에서 중요한가

**나쁜 보고서:**
```
성능 개선 결과 보고

구현 내용:
- DB 커넥션 풀 30→50
- 쿼리 타임아웃 5s→8s
- 캐시 TTL 300s→600s

결과:
- p99: 1200ms → 650ms (45.8%)
- TPS: 98 → 155 (+58%)
- 에러율: 0.5% → 0.08%

[CEO의 반응]
"...그래서 뭐가 좋다고?"
```

**좋은 보고서:**
```
주문 API 성능 개선 완료
→ 월간 추가 매출 3억 5천만 원 예상

주요 성과:
- 고객 만족도: 95명 중 1명의 느린 경험 제거
- 매출 증대: 전환율 2% → 6.5% 향상
- 비용 절감: 서버 비용 월 150만 원 절감

투자 대비:
- 개발 비용: 800만 원
- 연간 효과: 42억 원
- ROI: 5,250배
```

**차이점:**
- 나쁜 보고서: 기술자가 만족
- 좋은 보고서: 비즈니스 결정 가능

**성능 개선은 "비즈니스 문제 해결"이어야 의사결정을 얻습니다.**

---

## 😱 흔한 실수 (Before)

### 사례 1: 기술 용어만 있는 보고서
```
성능 개선 완료

변경 사항:
- Hikari DataSource 풀 크기 최적화
- JVM GC 튜닝 (-XX:+UseG1GC)
- Redis 캐시 계층 추가

결과:
- p99 latency: 1200ms → 650ms
- Throughput: 98 req/s → 155 req/s
- Error rate: 0.5% → 0.08%

PM 반응: "좋다고 치는데, 사실 뭐가 좋은 건지 모르겠어."
```

**문제점:**
- CEO/PM이 이해할 언어가 아님
- 비즈니스 가치 불명확
- 의사결정 불가

### 사례 2: 데이터 없는 주장
```
성능이 많이 개선되었습니다.

사용자들이 훨씬 만족할 것으로 예상됩니다.
우리의 경쟁력이 향상될 것입니다.

(구체적 수치 없음)
```

**문제점:**
- 정성적 주장만 있음
- 검증 불가능
- 의사결정에 사용할 수 없음

### 사례 3: Before/After 비교 없음
```
현재 성능:

p50: 180ms
p95: 420ms
p99: 650ms
에러율: 0.08%

[무엇과 비교하는가? 답: 모름]
```

**문제점:**
- 개선된 것인지 악화된 것인지 알 수 없음
- 비교 기준이 없음

### 사례 4: 그래프/표 생략
```
수치만 나열:
- p99: 1200ms → 650ms
- TPS: 98 → 155
- 에러율: 0.5% → 0.08%

[시각화 없으면]
- 직관적 이해 어려움
- 임팩트 저하
- 의사결정권자 설득력 약함
```

## ⚖️ 트레이드오프

### 메모리 증가 vs 성능 개선

| 항목 | 증가량 | 비용 | 효과 | 평가 |
|------|--------|------|------|------|
| DB 풀 | +20MB | 월 0원 | p99 -150ms | ✅ |
| 캐시 | +50MB | 월 50만 원 | p99 -100ms | ✅ |
| **총합** | +70MB | 월 50만 원 | p99 -250ms | ✅ |

**평가:** 월 50만 원의 추가 비용으로 p99 250ms 개선 = 월 56억 원 효과 대비 무시할 수준

### 관리 복잡도 증가

| 항목 | 추가 업무 | 시간 | 비용 |
|------|---------|------|------|
| 모니터링 | 주 1회 | 주 1시간 | 월 20만 원 |
| 튜닝 | 분기 1회 | 분기 8시간 | 월 10만 원 |
| **총합** | - | 월 3시간 | 월 30만 원 |

**평가:** 추가 관리 비용 월 30만 원 << 효과 월 56억 원

## 📌 핵심 정리

1. **Executive Summary 먼저 쓰라**
   - CEO/PM은 1분 안에 결정
   - 처음 3줄에 모든 중요 정보 담기

2. **비즈니스 언어로 번역하라**
   - "p99 550ms 개선" → "100명 중 느린 경험 95명 → 2명"
   - "에러율 0.42% 감소" → "월 4,200명의 이탈 방지"

3. **Before/After를 명확히 하라**
   - 표와 그래프로 시각화
   - 개선 정도를 직관적으로 이해 가능하도록

4. **ROI를 명시하라**
   - 투자 비용
   - 예상 효과
   - 회수 기간
   - 경영진이 의사결정 가능하도록

5. **주의사항과 가정을 명확히 하라**
   - 모든 가정 나열
   - 리스크 제시
   - 향후 모니터링 계획

## 🤔 생각해볼 문제

### Q1: CEO가 "그래서 꼭 해야 하는 건가?"라고 물어볼 때 어떻게 답할까?

```
성능 개선: 연 56억 원 효과
개발비: 800만 원

CEO: "그래도 우리 유지보수 팀이 바쁜데, 
꼭 지금 해야 하나? 다음 분기는?"

어떻게 설득할까?
```

<details>
<summary>해설 보기</summary>

**전략적 대답:**

1. **시간 가치 강조:**
   ```
   "지금 하면: 즉시 월 56억 원 효과
   다음 분기에 하면: 3개월 × 56억 원 = 168억 원 손실"
   ```

2. **기회비용 설명:**
   ```
   "개발 비용 800만 원보다
   3개월 지연 비용(168억 원)이 훨씬 크다"
   ```

3. **경쟁사 언급:**
   ```
   "네이버/당신/쿠팡은 이미 이 정도의 성능.
   우리가 같은 수준이 돼야
   고객 이탈을 막을 수 있다"
   ```

4. **우선순위 조정:**
   ```
   "개발팀의 다른 업무와 우선순위를 재검토해,
   이것보다 더 중요한 게 있나 확인하자"
   ```

**핵심:**
- CEO는 기회비용을 매우 중시함
- "800만 원 투자"보다 "168억 원 손실" 방지가 설득력 있음

</details>

### Q2: 개발팀이 "p99를 650ms까지만 할 수 있는데, 500ms 목표라고 했는데?"라고 할 때?

```
PM 요청: p99 < 500ms
개발팀 달성: p99 = 650ms
```

<details>
<summary>해설 보기</summary>

**데이터 기반 협상:**

1. **650ms의 비즈니스 가치 재산:**
   ```
   p99 650ms → 전환율 +4.5% → 월 27.5억 원
   p99 500ms → 전환율 +5.5% → 월 33.5억 원
   
   차이: 6억 원/월 = 72억 원/년
   추가 개발: 3주 × 500만 원 = 1,500만 원
   
   = 비용 대비 효과 긍정적?
   = 아니다. 150만 원 투자로 72억 원은 너무 큼!
   ```

2. **단계적 접근:**
   ```
   "먼저 650ms로 출시.
   3개월 뒤 실제 전환율 변화 측정.
   
   만약 목표에 못 미치면,
   그때 500ms 목표로 추가 개선하자.
   
   지금 추측으로 추가 3주 개발하는 것보다,
   실제 데이터로 판단하자."
   ```

**핵심:**
- 모든 거는 측정 후에 결정
- 과도한 완벽주의는 기회비용 큼

</details>

### Q3: 성능 보고서를 프레젠테이션할 때 어떤 순서로 말해야 할까?

```
1분 짧은 버전 vs 10분 상세 버전
각각 어떻게 구성할까?
```

<details>
<summary>해설 보기</summary>

**1분 버전 (Elevator Pitch):**

```
"주문 API 성능을 개선해서,
100명 중 극악의 경험을 하던 1명을 거의 제거했습니다.

결과:
- 고객 만족도 상승
- 월 27.5억 원 추가 매출 예상
- 투자 8백만 원 대비 7,000배 수익

이번 달에 시작하면 다음 달부터 효과를 볼 수 있습니다."
```

**5분 버전:**

```
1. Before/After 그래프 (30초)
   "이렇게 개선됐습니다"

2. 비즈니스 임팩트 (2분)
   - 매출: 월 27.5억 원
   - 고객 만족: 이탈 감소
   - 비용: 절감

3. 투자 대비 효과 (30초)
   - ROI: 7,000배
   - 회수 기간: 4시간

4. 다음 스텝 (1분)
   - 일정: 이번 달
   - 위험도: 낮음
   - 권장: 즉시 진행
```

**10분 버전 (상세 회의):**

```
1. 상황 분석 (2분)
   - Before: 뭐가 문제였나?
   - 비용: 월 얼마나 손실?

2. 개선 전략 (2분)
   - 어떻게 개선했나?
   - 기술 상세 (DB, 캐시 등)

3. 결과 분석 (3분)
   - Before/After 비교
   - 성능 수치
   - 안정성 개선

4. 비즈니스 임팩트 (2분)
   - 매출 분석
   - 비용 절감
   - ROI 계산

5. Q&A (1분)
```

**핵심:**
- 짧을수록: 결과만 (숫자 3개)
- 길수록: 프로세스도 (HOW/WHY)

</details>
