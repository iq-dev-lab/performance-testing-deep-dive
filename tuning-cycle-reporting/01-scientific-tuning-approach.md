# 01. 과학적 튜닝 접근법 — 하나씩 변경하는 이유

---

## 🎯 핵심 질문

"네 개의 설정값을 동시에 변경했는데 응답시간이 빨라졌어. 어느 설정이 효과 있었는지 알 수 있을까?"

만약 대답할 수 없다면, 당신은 이미 과학적 접근법을 벗어나 있습니다.

---

## 🔍 왜 이 개념이 실무에서 중요한가

성능 튜닝은 과학입니다. 가설을 세우고, 실험하고, 결과를 측정하고, 결론을 내리는 반복 과정입니다.

**변수 격리 원칙(Isolation of Variables)**을 위반하면:
- 어떤 변경이 효과 있었는지 알 수 없음
- 부작용이나 트레이드오프를 놓칠 수 있음
- 다른 환경에서 같은 결과가 나오지 않을 수 있음
- 롤백이 필요할 때 어느 것을 되돌려야 할지 불명확함

특히 팀에서 일할 때, 나중에 다른 개발자가 "왜 이렇게 설정했어?"라는 질문에 명확히 답하지 못하면 신뢰가 떨어집니다.

실제 사례:
- DB 커넥션 풀 크기를 30에서 50으로, 쿼리 타임아웃을 5초에서 10초로, 캐시 TTL을 300초에서 600초로 동시에 변경했다면?
- 어느 것이 p99 지표 개선에 기여했는지 알 수 없음
- 6개월 후 메모리 부족 문제가 생겨 롤백하려 할 때, 어떻게 롤백할 것인가?

## 😱 흔한 실수 (Before)

### 사례 1: 한번에 여러 개 변경
```
2024-03-15 10:00
- 변경: DB 풀 크기 30→50, 타임아웃 5s→10s, 캐시 TTL 300s→600s, 워커 스레드 10→20
- 결과: p99 3200ms → 800ms (75% 개선!)
- 결론: 성공!

[3개월 후]
메모리 부족 문제 발생. 뭐가 문제인지 모르겠음. 일단 DB 풀만 줄여볼까?
```

**문제점:**
- 75% 개선은 누구의 공적인가? 모든 변경?
- 만약 캐시 TTL 증가 때문이었다면, 나머지는 불필요한 리소스 낭비
- 어떤 변경이 트레이드오프를 일으켰는지 불명확

### 사례 2: 측정 없이 "경험상"
```
개발자 A: "우리 시스템이 느린 것 같은데..."
개발자 B: "알았어, 캐시를 3배 늘려보자. 일반적으로 도움이 돼."

[적용 후]
여전히 느림. 왜? 캐시가 병목이 아니었기 때문.
```

**문제점:**
- 가설이 없음 = 검증할 수 없음
- "일반적으로"는 당신의 시스템에 해당하지 않을 수 있음

### 사례 3: 튜닝 일지 미작성
```
팀원: "작년에 이거 왜 이렇게 설정했는데?"
당신: "음... 누군가 변경했을 거야?"

[결과]
누구도 이유를 모름. 비즈니스 요구사항 변경되었나? 성능 문제가 있어서?
```

## ✨ 올바른 접근 (After)

### 과학적 튜닝 사이클

```
1단계: 가설 수립
↓
2단계: 현재 상태 측정 (Baseline)
↓
3단계: 하나의 변경만 적용
↓
4단계: 재측정
↓
5단계: 결론 도출
↓
6단계: 결과 기록
↓
7단계: 다음 가설로 (1단계로 돌아가기)
```

### 예시: 올바른 튜닝 프로세스

**Experiment #1: DB 커넥션 풀 크기 최적화**

```
1. 가설: 
   "현재 DB 커넥션 풀(크기 30)이 병목이다. 
   풀 크기를 50으로 늘리면 동시 요청 처리가 개선되어 p99가 500ms 감소할 것이다."

2. Baseline 측정:
   - 부하: 동시 사용자 100명, 5분 실행
   - 명령어: k6 run load-test.js --vus 100 --duration 5m
   - 결과: p50=200ms, p95=600ms, p99=1200ms, 에러율=0.5%

3. 변경 적용:
   - 파일: application.yml
   - 변경: spring.datasource.hikari.maximum-pool-size: 30 → 50
   - (다른 설정은 손대지 않음!)

4. 재측정:
   - 동일한 부하 테스트 다시 실행
   - 결과: p50=200ms, p95=580ms, p99=1050ms, 에러율=0.4%

5. 결론:
   - p99: 1200ms → 1050ms (150ms = 12.5% 개선)
   - 에러율: 0.5% → 0.4% (0.1% 감소)
   - 결론: 효과 있음! 다음 단계로.

6. 기록:
   [아래의 마크다운 템플릿 사용]

7. 다음 가설:
   "이제 쿼리 타임아웃(5s)이 병목일 가능성이 높다. 
   이를 10s로 늘리면..."
```

## 🔬 내부 동작 원리

### 변수 격리가 필요한 이유

**시스템은 비선형(Non-Linear)이다:**
```
풀 크기 조정 효과 (단독):    p99 -150ms
타임아웃 조정 효과 (단독):   p99 -200ms
캐시 조정 효과 (단독):       p99 -100ms

세 개 동시 조정:
예상: -150ms -200ms -100ms = -450ms
실제: -600ms (상호작용 효과)

또는:
예상: -450ms
실제: -300ms (간섭 효과)
```

**왜 이런 일이 일어나는가?**

1. **경합(Contention) 효과:**
   - 풀 크기 증가 → 더 많은 동시 쿼리 가능
   - 타임아웃 증가 → 각 쿼리가 더 오래 기다림
   - → 전체 시스템의 처리량은 비선형으로 변함

2. **리소스 한계:**
   - CPU, 메모리, I/O 대역폭은 유한함
   - 한 설정을 개선하면 다른 리소스가 병목이 될 수 있음

3. **악순환(Cascading Effects):**
   - 캐시 증가 → 메모리 증가 → GC 일시정지 증가 → 응답시간 증가

### 선택적 가설 수립의 중요성

**좋은 가설:**
```
"현재 p99=1200ms이고, 
APM 도구 분석 결과 DB I/O 대기 시간이 전체의 60%를 차지한다. 
DB 커넥션 풀을 현재 30에서 50으로 늘리면, 
동시성이 개선되어 p99가 200ms 이상 감소할 것이다."
```
✅ 구체적, 증거 기반, 측정 가능

**나쁜 가설:**
```
"성능이 느린 것 같다. 캐시를 크게 해보자."
```
❌ 증거 없음, 측정 기준 불명확, 성공 기준 불명확

## 💻 실전 실험

### Step 1: Baseline 측정

```bash
# 1. 테스트 스크립트 준비 (load-test.js)
cat > load-test.js << 'EOF'
import http from 'k6/http';
import { check, sleep } from 'k6';

export const options = {
  vus: 100,        // 동시 사용자 100명
  duration: '5m',  // 5분 동안 실행
  thresholds: {
    http_req_duration: ['p(99)<2000'],  // p99가 2초 이상이면 알림
  },
};

export default function () {
  const res = http.get('http://api.example.com/products/1');
  
  check(res, {
    'status is 200': (r) => r.status === 200,
    'response time < 1s': (r) => r.timings.duration < 1000,
  });
  
  sleep(1); // 각 요청 사이에 1초 대기
}
EOF

# 2. Baseline 측정 실행
k6 run load-test.js --out json=baseline.json

# 3. 결과 확인
cat baseline.json | jq '.metrics'
```

**출력 예시:**
```json
{
  "p50": 200,
  "p95": 600,
  "p99": 1200,
  "error_rate": 0.005
}
```

### Step 2: 가설 기록

```markdown
## Experiment #1: DB 커넥션 풀 크기

**가설:**
현재 DB 풀 크기(30)가 병목이다. 50으로 늘리면 p99가 200ms 이상 감소할 것이다.

**근거:**
- APM 분석: DB I/O 대기 시간 = 60% of total duration
- 동시 요청: 100 VU × 1 req/sec = 100 requests/sec
- 현재 풀: 30 커넥션
- → 평균 커넥션당 3.3개 요청 대기 중

**측정 항목:**
- p50, p95, p99 응답시간
- TPS (처리량)
- 에러율
- GC 일시정지 시간

**성공 기준:**
- p99: 1200ms → 1000ms 이상 개선
- 에러율: 현재 0.5% 이하 유지
```

### Step 3: 변경 적용

```bash
# 1. 설정 변경
sed -i 's/maximum-pool-size: 30/maximum-pool-size: 50/' application.yml

# 2. 변경 사항 확인
grep maximum-pool-size application.yml

# 3. 애플리케이션 재시작
./restart-app.sh

# 4. 헬스체크 (변경 사항이 반영될 때까지 대기)
curl -s http://localhost:8080/health | jq .
```

### Step 4: 재측정 및 비교

```bash
# 1. 동일한 부하 테스트 실행
k6 run load-test.js --out json=after-experiment1.json

# 2. 비교 스크립트 (compare.js)
cat > compare.js << 'EOF'
const baseline = require('./baseline.json');
const after = require('./after-experiment1.json');

function parseResult(metrics) {
  return {
    p50: metrics.find(m => m.name === 'p50')?.value || 0,
    p95: metrics.find(m => m.name === 'p95')?.value || 0,
    p99: metrics.find(m => m.name === 'p99')?.value || 0,
    errorRate: metrics.find(m => m.name === 'error_rate')?.value || 0,
  };
}

const baselineData = parseResult(baseline.metrics);
const afterData = parseResult(after.metrics);

console.log('=== Baseline vs After ===');
console.log(`p99: ${baselineData.p99}ms → ${afterData.p99}ms (${((baselineData.p99 - afterData.p99) / baselineData.p99 * 100).toFixed(1)}% 개선)`);
console.log(`Error Rate: ${(baselineData.errorRate * 100).toFixed(2)}% → ${(afterData.errorRate * 100).toFixed(2)}%`);
EOF

node compare.js
```

## 📊 성능 비교

### 실제 사례: 3단계 튜닝 사이클

| 지표 | Baseline | Exp #1<br>(풀 크기 30→50) | Exp #2<br>(+타임아웃 5s→8s) | Exp #3<br>(+캐시 300s→600s) |
|------|----------|------------------------|---------------------------|---------------------------|
| **p50** | 200ms | 190ms | 185ms | 180ms |
| **p95** | 600ms | 550ms | 480ms | 420ms |
| **p99** | 1200ms | 1050ms | 780ms | 650ms |
| **TPS** | 98 req/s | 112 req/s | 135 req/s | 155 req/s |
| **Error Rate** | 0.50% | 0.40% | 0.15% | 0.08% |
| **Memory** | 512MB | 535MB | 545MB | 580MB |

### 분석

1. **Experiment #1 효과:**
   - p99: 1200ms → 1050ms (12.5% 개선)
   - 원인: 동시 커넥션 증가로 병렬성 향상

2. **Experiment #2 효과:**
   - p99: 1050ms → 780ms (25.7% 추가 개선)
   - 원인: 느린 쿼리도 정상 완료 가능

3. **Experiment #3 효과:**
   - p99: 780ms → 650ms (16.7% 추가 개선)
   - 원인: 데이터베이스 조회 빈도 감소

**누적 개선:**
- 전체: p99 1200ms → 650ms (45.8% 개선)
- 각 변경의 기여도 명확함

## 📌 핵심 정리

1. **한 번에 하나의 변경만 적용하라**
   - 변수 격리 원칙
   - 각 변경의 효과를 측정 가능하게 함

2. **항상 Baseline부터 시작하라**
   - "현재 상태"를 정확히 기록
   - 개선도를 정량적으로 계산 가능

3. **가설을 명시적으로 기록하라**
   - 왜 이 변경을 시도하는가?
   - 성공 기준이 무엇인가?
   - 근거가 무엇인가?

4. **모든 결과를 튜닝 일지에 기록하라**
   - 6개월 후 "왜 이렇게 설정했어?"라는 질문에 답할 수 있음
   - 비슷한 문제 재발 시 참고 자료

5. **트레이드오프를 문서화하라**
   - p99 300ms 개선 vs 메모리 +50MB 증가?
   - 비즈니스 요구사항과 함께 검토

### Q2: 언제까지 튜닝을 계속해야 할까?

```
p99 개선:
- Baseline: 1200ms
- Exp #1: 1050ms (12.5% ✅)
- Exp #2: 780ms (25.7% ✅)
- Exp #3: 650ms (16.7% ✅)
- Exp #4: 640ms (1.5% 🤔)
- Exp #5: 635ms (0.8% 🤔)
```

<details>
<summary>해설 보기</summary>

**수익 감소 법칙(Law of Diminishing Returns):**

초반 개선은 크고 쉬움. 나중으로 갈수록 작아지고 어려워짐.

**언제 멈춰야 할까?**

1. **비즈니스 목표 달성 시**
   - 목표: p99 < 700ms → 달성 (Exp #3)
   - 더 이상 튜닝할 이유 없음

2. **노력 대비 효과가 너무 작을 때**
   - Exp #4, #5는 각각 1-2%만 개선
   - 새로운 버그 위험 vs 0.8% 개선?
   - → 비용 대비 효과 부족

3. **다른 병목이 더 심할 때**
   - p99는 이미 650ms로 충분함
   - 다른 지표 (에러율, TPS)가 문제면?
   - → 다른 곳에 자원 투자

**의사결정 기준:**
- 효과 >= 5% → 계속 진행
- 효과 1-5% → 검토 후 진행
- 효과 < 1% → 스톱, 다른 문제 해결

</details>

### Q3: 팀 동료가 "나는 그냥 5개 설정을 한 번에 바꿨는데 좋았어"라고 주장한다면?

```
"왜 하나씩 해? 
한 번에 5개를 바꾸면 빨라지는 거 확인했는데..."
```

<details>
<summary>해설 보기</summary>

**이렇게 답하세요:**

1. **과학적 근거 제시**
   ```
   "맞아, 빨라졌을 수 있어. 
   근데 어느 설정이 효과 있었는지 모르지 않아?
   
   예를 들어:
   - 캐시 증가만으로도 80% 효과 났다면? 
     → 나머지 4개는 불필요하고 메모리만 낭비
   - 메모리 문제가 생겨서 롤백해야 한다면?
     → 어떤 걸 되돌려야 하나?"
   ```

2. **장기적 관점**
   ```
   "앞으로 6개월 마다 성능 튜닝할 때,
   지금 왜 이 설정으로 했는지 기록해야
   다음번에도 적용할 수 있지 않을까?
   
   기록 없이 '대략 잘 됐다'는 느낌만으로
   매번 새로 시작할 거야?"
   ```

3. **실제 트레이드오프 분석**
   ```
   "이번엔 빨라졌을 수 있어.
   근데 메모리, CPU, 관리 복잡도 변화는?
   
   한 번에 5개를 바꾸면:
   - 메모리 50% 증가 가능성
   - 버그 위험 증가
   - 문제 생겼을 때 원인 파악 불가
   
   하나씩 하면:
   - 각 단계의 효과 정확히 측정
   - 진짜 필요한 것만 적용
   - 문제 생기면 쉽게 롤백"
   ```

4. **프로세스의 가치**
   ```
   "성능 튜닝은 일회성이 아니야.
   앞으로 계속할 거고,
   더 복잡한 시스템에도 해야 할 거야.
   
   지금부터 프로세스를 정립해두면,
   나중에 팀 전체의 생산성이 올라가."
   ```

**설득 불가능하면:**
- 한 번에 5개 변경 후 분석
- 3-4주 뒤 성능 악화 발생 시 (높은 확률)
- "이제 어느 걸 롤백할까?" 상황에서
- 자연스럽게 일관된 프로세스의 필요성 깨달음

</details>
